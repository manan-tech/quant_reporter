# quant_reporter v2.1 — Program Roadmap

**Date:** 2026-05-31
**Branch:** `v2.1`
**Status:** Decomposition approved; SP0 in design.
**Evidence base:** `2026-05-31-v2.1-audit-research-raw.json` (full 13-agent audit + research, verbatim).

This is the north-star for v2.1. It is **not** an implementation spec — each sub-project (SP0–SP4)
gets its own `*-design.md` spec → plan → build cycle. This doc records the vision, the decisions,
the decomposition, and the findings that justify them.

---

## 1. Vision & posture

`quant_reporter` becomes an **institutional, primitives-first** portfolio library.

- **Default = transparent, composable building blocks.** Every primitive is a pure function that
  takes pandas/numpy in and returns plain objects (Series/DataFrame/dict) out. No black box.
- **Opinions live only behind an opt-in `recommend=True` flag.** Half-Kelly, a 10% vol target,
  CPPI `m=5`, a default cost model — these are *recommendations*, never baked into the primitives.
- **A few canonical example strategies ship** so retail users learn to compose primitives themselves.
- **Reports are one consumer of the primitives, not the source of truth.**

Target users span retail → hedge-fund PM/quant. The primitives serve the institutional end (control,
auditability); the opt-in recommendation layer + example strategies serve the retail end (guidance).

## 2. Decisions (locked with the user)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Posture | Institutional-first primitives; opt-in `recommend=True`; ship example strategies | Serves both audiences without a black box; maps onto existing `compute_*`/`create_*` split |
| Build order | **Core first**, then strategies | Everything plugs into the analytics core; building on today's divergent base bakes bugs deeper |
| Dual-basis fix | **Fix now, canonical basis** | Reports currently show inconsistent numbers under the same labels; fix is a documented 2.1 behavior change |
| Package layout | **Keep flat** (`src/quant_reporter/*.py`, re-export from `__init__`) | Preserves the `from quant_reporter import X` contract and the one-screen-per-module house style |
| Dependencies | **No new deps** | sklearn/scipy/statsmodels/requests already pinned; CVaR uses `scipy.linprog` (HiGHS), never cvxpy |
| Text layer (item 2) | Templated default + pluggable LLM hook | Matches "dynamic, not LLM — but optionally LLM"; no hard dependency |

## 4. SP0 — Analytics core (item 1): what the audit found

The audit found **latent correctness bugs**, not just duplication. Highest-impact first:

1. **Dual return basis (the #1 bug).** Optimizers/frontier/risk-contribution/Monte-Carlo use
   **log-return** moments (`opt_core.get_optimization_inputs:69`, `get_portfolio_stats:18-20`),
   while the metrics grid uses **simple `pct_change`** vol/Sharpe (`metrics.py:66,84`) and
   **geometric** CAGR (`:80`). The same strategy shows a different return/vol/Sharpe in the
   efficient-frontier star vs. the metrics card.
2. **`rebalance_freq` is a silent no-op.** Plumbed `create_combined_report:61` → `build_context:85`
   → `ctx` (`report_context.py:40,181`) but **no `compute_*` reads it**.
   `simulate_rebalanced_portfolio` is imported only in `__init__.py:108`. Requesting a rebalance
   frequency changes nothing in any output.
3. **Monte Carlo engine is forked.** `monte_carlo_report.py:13-185` is a ~90% copy of
   `monte_carlo.py:11-239`; both live; they **will diverge**. No RNG seed → every run differs.
4. **Growth-of-$1 reimplemented 5+ ways, drawdown 4×**, on inconsistent bases — the plotted
   underwater trough can disagree with the headline Max Drawdown.
5. **`calculate_metrics` returns formatted strings** (`"12.3%"`) that `validation_report._to_num`
   parses back to floats — precision loss + a documented past bug.
6. **Factor/attribution gaps.** Brinson measures against **equal weights over the portfolio's own
   tickers** (`factor_report.py:219`), not `ctx.friendly_benchmark`; two different OLS engines
   (statsmodels static vs. hand-rolled rolling on raw returns); rfr never threaded in.
7. **Combined report is not a strict superset.** MC params locked to 5000/252/$10k
   (`combined_report.py:124`); delegate failures silently swallowed (`:99-126`); dead `desc=` kwarg.

### Proposed single source of truth
A **headless analytics core** = the sole producer of every quantity, returning **numerics on one
basis**, with reports as **pure assemblers** that never recompute. Canonical homes:

| Concern | Canonical home | Notes |
|---------|----------------|-------|
| Portfolio returns / growth-of-$1 / terminal | one returns helper | absorbs `get_portfolio_price`, the `(1+r).cumprod()` growth, attribution's `.prod()-1`, **and** `simulate_rebalanced_portfolio` so `rebalance_freq` is honored |
| Drawdown | `calculate_max_drawdown` returns **curve + scalar** | every drawdown plot feeds from it |
| Moments | `get_optimization_inputs` only | denoise + frequency-aware annualization threaded; metrics consume these, not re-derive |
| Risk metrics | `calculate_metrics` returns **numerics** + separate formatter | one rolling-Sharpe, VaR/CVaR with explicit horizon-vs-daily labels |
| Monte Carlo | merge the fork into one **seeded** engine | passes `ctx.risk_free_rate`; supports stress + actual-path overlay |
| Risk-free rate / factor regression | one rfr resolution, one OLS engine | single fallback; one excess-return convention for static + rolling |

(Full metric→producers matrix with divergent flags is in the raw appendix, `auditSynth.metric_matrix`.)

## 5. SP1/SP2 — Strategy roadmap (item 4)

**The one structural unlock:** `validation_report.run_rolling_windows` already builds per-window
optimal weights (`:102-118`) and **throws them away**, keeping only OOS Sharpe. Returning them as a
`target_weight_schedule` is the keystone — it is the missing input to `simulate_strategy`, which feeds
turnover → costs → tax → tracking → OOS comparison. That single change converts the library from
"optimize + one-shot simulate" to a **walk-forward, cost-aware, honestly-evaluated backtest engine**.

**New flat modules** (mirror `advanced_optimizers.py` house style): `backtest.py`, `performance_stats.py`,
`sizing.py`, `signals.py`, `robust_estimators.py`, `factor_tilts.py`.
**Extend:** `opt_core.py` (CVaR via `scipy.linprog`, `risk_contributions`, `optimize_risk_budget`),
`validation_report.py` (expose the weight schedule).

**Hubs that unlock the most:** `simulate_strategy` (every backtest/tax/tracking/OOS primitive plugs
into it) and `volatility_target_positions` (the entire tactical-signals family sizes through it).
`ledoit_wolf_covariance` is the highest value-per-line: one shrinkage call hardens the cov input of
all seven existing optimizers + Black-Litterman.

### Canonical example strategies to ship
1. `example_walk_forward_backtest.py` — rolling windows → schedule → `simulate_strategy(cost_model)`
   → `compare_strategies_oos` with DSR/PSR (the flagship "honest OOS" demo).
2. `example_robust_inputs.py` — Ledoit-Wolf + James-Stein → `find_optimal_portfolio`, vs. raw inputs.
3. `example_trend_following.py` — TS-momentum + MA-crossover ensemble → vol-target → simulate (CTA sleeve).
4. `example_factor_tilt.py` — cross-sectional momentum / characteristic tilt → simulate → factor
   regression to confirm realized loadings (closes the factor loop).
5. `example_risk_overlays.py` — vol-targeting + CPPI stacked on a base optimizer, drawdown before/after.

## 6. SP3 — Per-asset info layer (item 2)

A standardized per-holding structure (DataFrame/dict primitive), consumed by reports:

- **Computed-from-price analytics** — per-asset return, vol, Sharpe/Sortino, beta/alpha vs benchmark,
  max drawdown, correlation, contribution to portfolio risk/return.
- **Per-asset factor exposures** — loadings on the Fama-French factors (reuse the factor engine).
- **Allocation context** — current weight, weight vs benchmark, sector bucket; suggested delta when
  `recommend=True`.
- **Reference/fundamental data** — name, sector, market cap, P/E, dividend yield, 52-wk range from
  yfinance `.info`; **gracefully degrading** (best-effort, never breaks a report).
- **Text layer** — templated/deterministic narration from metric thresholds by default; **pluggable
  LLM hook** (user supplies a callable) for richer prose. No hard LLM dependency.

## 7. SP4 — Recommendation layer (item 3)

Opt-in `recommend=True`. A dedicated module producing **structured, transparent** recommendation
objects, each carrying the metric/rationale that drove it:

- **Recommended target weights** — from a chosen objective (or compare optimizers and pick).
- **Rebalance trade list** — current vs target → buy/sell deltas, threshold logic, turnover/cost
  estimate (uses the SP1 backtest engine).
- **Risk alerts & limit breaches** — vol above target, drawdown/concentration/sector-cap/factor-drift.
- **Strategy-comparison verdict** — rank strategies via the backtest engine; recommend one with evidence.

## 8. Cross-cutting conventions (enforced across all SPs)

- **Numerics, never display strings.** Primitives return floats/Series/dicts; a separate formatter
  handles `"%"`/2dp. Kills the `validation_report._to_num` round-trip.
- **One annualization constant.** `get_optimization_inputs` already does `×252`; every new
  returns-in/annualized-out primitive annualizes once with the same factor (frequency-aware).
  Annualization double-count is a top silent-bug risk.
- **Weights are `np.ndarray` (ticker order) in optimizers, `dict` in rebalancing/reports.**
  `simulate_strategy` accepts **both** (dict one-shot, DataFrame schedule) — load-bearing back-compat.
- **Look-ahead safety is the dominant correctness risk.** Every rolling stat uses trailing-only
  windows; every signal is lagged ≥1 period; schedule rows decided on data up to *d−1*, traded at *d*.
  Bake into **hypothesis** property tests (already in the `[test]` extra).
- **Back-compat watch.** `simulate_rebalanced_portfolio` returns a `(Series, DataFrame)` tuple and is
  the only rebalancing export — **do not change its signature**; `simulate_strategy` is the additive superset.
- **`recommend=True` is the only home for opinions.** Primitives stay opinion-free with explicit params.
